# Test-Driven Development

> 原始來源：`.agent/skills/_vendor/superpowers/skills/test-driven-development/SKILL.md`

## 觸發時機

實作任何功能或 bugfix 時，在寫生產程式碼**之前**必須執行。

## 鐵律

**NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST**

先寫測試前的程式碼？刪除。重新開始。沒有例外。

## RED-GREEN-REFACTOR 循環

### RED — 寫失敗的測試

- 一個測試只測一個行為。
- 清晰命名：描述行為而非實作。
- 使用真實程式碼，mocks 僅限必要。

### 驗證 RED — 看它失敗

確認：
- 測試失敗（非錯誤）。
- 失敗訊息符合預期。
- 因功能缺失而失敗（非打字錯誤）。

若測試通過 → 測試的是現有行為，修正測試。

### GREEN — 寫最少程式碼

只寫足夠通過測試的程式碼。不添加功能、不重構其他程式碼、不「改進」超過測試範圍。

### 驗證 GREEN — 看它通過

確認：
- 測試通過。
- 其他測試仍通過。
- 輸出乾淨（無錯誤、警告）。

### REFACTOR — 清理

綠色後：移除重複、改進命名、提取輔助函數。保持測試綠色，不添加行為。

## 常見藉口與現實

| 藉口 | 現實 |
|------|------|
| 太簡單無須測試 | 簡單程式碼也會壞，測試 30 秒 |
| 我之後補測試 | 事後測試立即通過，證明不了什麼 |
| 已手動測試 | 手動不可重現、無記錄、壓力下會遺漏 |
| 刪除 X 小時工作很浪費 | 沉沒成本謬誤，保留未驗證程式碼是技術債 |
| TDD 減速 | TDD 比事後除錯快 |

## 紅旗 — 停止並用 TDD 重新開始

- 程式碼在測試之前寫了。
- 測試寫完立即通過。
- 「就這一次」不走 TDD。
- 「保留現有程式碼作參考」（那就不是 TDD）。

## 驗證檢查清單

- [ ] 每個新函數/方法有測試
- [ ] 每個測試在實作前失敗
- [ ] 失敗原因是功能缺失（非打字錯誤）
- [ ] 寫最少程式碼通過測試
- [ ] 所有測試通過
- [ ] 輸出乾淨
- [ ] mocks 僅限必要
- [ ] 邊界情況與錯誤已涵蓋
