# Writing Plans

> 原始來源：`.agent/skills/_vendor/superpowers/skills/writing-plans/SKILL.md`

## 觸發時機

有設計規格或多步工作需求時，在動手寫程式碼**之前**使用。

## 核心原則

寫計畫時假設執行者對程式碼庫零背景且判斷力有限。記錄一切：每工作觸及哪些檔案、完整程式碼、測試、如何驗證。

## 計畫結構

### 檔案結構先行

在定義工作前，先映射所有將建立或修改的檔案與各自職責。設計清晰邊界與介面。

### 工作粒度：咬大小步驟

每步是一個動作（2-5 分鐘）：

```markdown
### 工作 N：[元件名]

**檔案：**
- 建立：`exact/path/to/file.py`
- 修改：`exact/path/to/existing.py:123-145`
- 測試：`tests/exact/path/to/test.py`

- [ ] **步驟 1：寫失敗的測試**
（完整測試程式碼）

- [ ] **步驟 2：執行測試驗證失敗**
（精確指令 + 預期輸出）

- [ ] **步驟 3：寫最少實作**
（完整實作程式碼）

- [ ] **步驟 4：執行測試驗證通過**
（精確指令 + 預期輸出）

- [ ] **步驟 5：提交**
（精確 git 指令）
```

## 禁止佔位符

以下是計畫失敗，絕不可出現：

- 「TBD」、「TODO」、「之後實作」
- 「添加適當錯誤處理」（未附程式碼）
- 「為上述寫測試」（未附測試程式碼）
- 「類似工作 N」（必須重複完整內容）
- 參考任何工作中未定義的類型或函數

## 自我審核

計畫完成後執行：

1. **規格涵蓋**：每個規格需求都能指向對應的工作？列出缺口。
2. **佔位符掃描**：搜尋上述禁止模式，修復。
3. **類型一致**：後續工作引用的函數名稱、簽名、屬性名是否與前面定義一致？

發現問題直接修復，不需再跑一輪審核。

## 執行銜接

計畫儲存後，提供兩種執行選項：

1. **Subagent-Driven**（推薦）：使用 `subagent-driven-development` skill，每工作派遣新 subagent + 兩階段審核。
2. **內聯執行**：在當前對話中逐步執行。

## 本地化調整

- 計畫文件使用台灣正體中文撰寫。
- 儲存位置依專案慣例，預設 `docs/plans/YYYY-MM-DD-<feature-name>.md`。
- 與現有 `AGENT_PLAN.md` 機制的關係：AGENT_PLAN 記錄高層待辦，writing-plans 產出的是單一待辦項的詳細執行計畫。
- 提交訊息遵循 `git_usage` rule。
