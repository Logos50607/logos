# Systematic Debugging

> 原始來源：`.agent/skills/_vendor/superpowers/skills/systematic-debugging/SKILL.md`

## 觸發時機

遇到任何 bug、測試失敗或非預期行為時，在提出修復**之前**必須先執行本 skill。

## 鐵律

**NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST**

症狀修復是失敗。總是在嘗試修復前找出根本原因。

## 四個階段（必須按順序完成）

### 階段一：根本原因調查

1. 仔細讀取完整錯誤訊息（不跳過警告）。
2. 重現問題：記錄步驟與可靠性。
3. 檢查最近變更：`git diff`、新依賴、環境變化。
4. 在多元件系統中蒐集證據：為每個元件邊界添加診斷。
5. 追蹤資料流：當錯誤在深層呼叫堆疊時，向後追蹤至原始觸發。

### 階段二：模式分析

1. 尋找有效例子：相同程式碼庫中類似的有效代碼。
2. 對比參考實作。
3. 找出所有差異（無論多小）。
4. 理解依賴關係。

### 階段三：假說與測試

1. 形成單一假說：清晰陳述為什麼相信這是根本原因。
2. 最小化測試：一次改一個變數。
3. 在繼續前驗證。
4. **不知道就說不知道**，不要假裝。

### 階段四：實作修復

1. 建立失敗的測試案例。
2. 實作單一修復。
3. 驗證修復有效。
4. 若修復失敗：
   - 嘗試 < 3 次 → 回到階段一。
   - 嘗試 ≥ 3 次 → **停止，質疑架構**，與使用者討論。

## 3+ 修復失敗：質疑架構

當每個修復揭露新的耦合或在其他地方產生新症狀時，問題不在修復本身而在基礎架構。此時必須停止並與使用者討論。

## 紅旗 — 立即停止並遵循流程

- 「快速修復待查」
- 「試試改 X 看看」
- 「添加多個變更一起測試」
- 「可能是 X」（未驗證）
- 在追蹤資料流前提出解決方案

## 本地化調整

- 與現有 `development_guidelines` rule 的測試驅動要求搭配：階段四的「建立失敗測試案例」對齊 TDD 原則。
- 與 `agent_behavior` rule 的重複指令限制搭配：3 次失敗後質疑架構，而非繼續循環。
